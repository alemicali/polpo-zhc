## Retry Flow

When a task fails assessment, Polpo uses a multi-stage retry system:

```
Assessment fails
    │
    ├── Fix attempts remaining?
    │   └── YES → Fix Phase (targeted corrections)
    │       └── Re-assess → passed? DONE
    │       └── Still failing → next fix attempt
    │
    ├── Fix attempts exhausted?
    │   └── Full Retry (start from scratch)
    │       └── Re-assess → passed? DONE
    │       └── Still failing → next retry
    │
    └── All retries exhausted?
        └── Check escalation policy
            ├── escalateAfter not reached → FAILED
            └── escalateAfter reached → switch agent/model, retry
```

## Fix Phase

The fix phase sends **targeted feedback** to the agent instead of starting over:

```
Agent receives:
- Original task description
- Per-dimension scores with reasoning
- Specific suggestions for improvement
- "Fix the issues above without rewriting from scratch"
```

This is faster and cheaper than a full retry because the agent builds on existing work.

### Configuration

```yaml
settings:
  maxFixAttempts: 2    # fix attempts per review cycle (default: 2)
```

After `maxFixAttempts` fix phases fail, the task falls back to a full retry.

## Full Retry

A full retry resets the task to `pending` and starts execution from scratch:

- Task status: `failed` → `pending`
- `retries` counter incremented
- Phase: `execution` (fresh start)
- `originalDescription` preserved for reference

### Configuration

```yaml
settings:
  maxRetries: 3    # full retry attempts (default: 3)
```

Per-task override:

```yaml
tasks:
  - title: Critical task
    maxRetries: 5
```

## Escalation

When a task keeps failing, you can escalate to a different agent or model:

```yaml
tasks:
  - title: Complex refactoring
    agent: junior-dev
    retryPolicy:
      escalateAfter: 2        # after 2 failures
      fallbackAgent: senior-dev  # switch to this agent
      escalateModel: claude-opus-4-6  # or upgrade the model
```

### How It Works

1. Task assigned to `junior-dev`
2. Fails assessment after fix + retry cycles
3. After `escalateAfter` failures, task is reassigned to `senior-dev`
4. Senior dev gets the full context: original description, previous attempts, assessment feedback
5. If `escalateModel` is set, the model is upgraded for the next attempt

### Default Escalation

Set a default policy for all tasks:

```yaml
settings:
  defaultRetryPolicy:
    escalateAfter: 3
    fallbackAgent: senior-dev
```

## Phase Tracking

Each task tracks its current phase:

| Phase | Description |
|-------|-------------|
| `execution` | Initial or retry execution |
| `review` | Assessment in progress |
| `fix` | Targeted correction from feedback |
| `clarification` | Agent asked a question, waiting for answer |

You can observe phase changes via the `task:transition` event and the task's `phase` field.

## Feedback Content

Assessment feedback sent to agents includes:

```
## Assessment Failed

### Dimension Scores:
- correctness: 2/5 (weight: 0.35)
  "The API endpoint returns 200 for invalid input instead of 400"
- completeness: 3/5 (weight: 0.30)
  "Missing the DELETE endpoint specified in requirements"
- code_quality: 4/5 (weight: 0.20)
  "Code is well-structured but lacks input validation"
- edge_cases: 2/5 (weight: 0.15)
  "No handling for empty request body or missing fields"

### Global Score: 2.7/5 (threshold: 3.0)

Fix the issues above. Focus on correctness and edge_cases.
```

<Aside>
The `originalDescription` field preserves the initial task description before any retry modifications. This ensures the agent always sees the original requirements even after multiple retry cycles.
</Aside>
