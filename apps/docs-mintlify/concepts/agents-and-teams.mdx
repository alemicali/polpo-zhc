Every Polpo project has a **team** — a named group of agents that do the actual work. You define them once in your config, and Polpo assigns tasks to them, tracks their activity, and cleans up when they're done.

## Agents

An agent is a named worker with an adapter (how it runs), an optional model, and optional constraints on what tools and files it can access.

```json
{
  "team": {
    "name": "my-project",
    "agents": [
      {
        "name": "backend-dev",
        "adapter": "claude-sdk",
        "role": "developer",
        "model": "claude-sonnet-4-20250514",
        "allowedTools": ["read", "write", "edit", "bash", "glob", "grep"],
        "allowedPaths": ["./src", "./tests"],
        "maxTurns": 150,
        "systemPrompt": "You are a senior backend developer. Focus on clean, tested code.",
        "skills": ["typescript-patterns"]
      },
      {
        "name": "frontend-dev",
        "role": "developer",
        "model": "anthropic:claude-sonnet-4-20250514",
        "allowedPaths": ["./src/ui", "./public"]
      }
    ]
  }
}
```

### Agent Config Reference

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `name` | `string` | *required* | Unique agent identifier |
| `adapter` | `string` | built-in engine | `"claude-sdk"` or any registered adapter |
| `role` | `string` | — | Human-readable role description |
| `model` | `string` | — | `"provider:model"` or bare model ID |
| `allowedTools` | `string[]` | all 7 tools | Which coding tools this agent can use |
| `allowedPaths` | `string[]` | project root | Filesystem sandbox — agent can only access these directories |
| `mcpServers` | `object` | — | MCP server connections (see [Tools & MCP](/features/tools-and-mcp)) |
| `systemPrompt` | `string` | — | Custom system prompt prepended to the default orchestration contract |
| `skills` | `string[]` | — | Named skills to load (markdown files with instructions) |
| `maxTurns` | `number` | `150` | Maximum tool-use turns before the agent is stopped |
| `maxConcurrency` | `number` | — | Per-agent concurrency limit |

### Adapters

Adapters control *how* an agent runs. Polpo ships with two:

<Tabs>
  <Tab title="Built-in Engine (default)">
    If you omit the `adapter` field, Polpo uses its **native engine** — a multi-provider LLM runtime powered by [pi-agent-core](/https://github.com/nicepkg/pi-agent-core). It supports 18+ providers (OpenAI, Anthropic, Google, Groq, Mistral, Ollama, etc.) and runs in-process.

    ```json
    {
      "name": "dev-1",
      "model": "anthropic:claude-sonnet-4-20250514"
    }
    ```

    The model format is `provider:model-id`. If you omit the provider prefix, Polpo resolves it from your configured providers.
  </Tab>

  <Tab title="Claude SDK">
    The `claude-sdk` adapter uses Anthropic's official [Claude Agent SDK](/https://github.com/anthropic/claude-agent-sdk). It runs with full tool permissions, persistent sessions, and MCP support.

    ```json
    {
      "name": "claude-agent",
      "adapter": "claude-sdk",
      "model": "claude-sonnet-4-20250514"
    }
    ```
  </Tab>
</Tabs>

You can also **write your own adapter** — any object implementing the `AgentAdapter` interface. See the [Custom Adapter](/reference/custom-adapter) guide.

### How Agents Run

Each agent task runs as a **detached Node.js subprocess** (the "runner"). This is what makes Polpo crash-resilient:

1. Orchestrator spawns a runner process per task
2. Runner loads the adapter, creates the agent, and starts working
3. Runner polls activity every 1.5s and writes it to the store
4. If the orchestrator crashes, runners keep going — restart the orchestrator and it reconnects to live PIDs
5. When the task is done, the runner writes the result and exits

The orchestrator's 5-second tick loop monitors all active runners: checking heartbeats, enforcing timeouts, and collecting results.

## Teams

A team is just a named container for agents:

```json
{
  "team": {
    "name": "api-builders",
    "description": "Team for the REST API project",
    "agents": [
      { "name": "dev-1", "role": "developer" },
      { "name": "dev-2", "role": "developer" },
      { "name": "reviewer", "role": "reviewer", "model": "anthropic:claude-sonnet-4-20250514" }
    ]
  }
}
```

You manage teams through the orchestrator:

```typescript
// Get the current team
const team = orchestrator.getTeam();

// Add an agent at runtime
orchestrator.addAgent({
  name: "specialist",
  role: "developer",
  model: "anthropic:claude-sonnet-4-20250514",
});

// Remove an agent
orchestrator.removeAgent("specialist");

// Rename the team
orchestrator.renameTeam("new-name");
```

Or via the TUI, which has built-in commands for team management.

## Volatile Teams

Sometimes a plan needs agents that don't exist in your main config. **Volatile teams** let a plan define its own temporary agents that exist only for the duration of that plan.

```json
{
  "group": "data-migration",
  "team": [
    {
      "name": "db-specialist",
      "role": "database engineer",
      "model": "anthropic:claude-sonnet-4-20250514",
      "systemPrompt": "You are a PostgreSQL migration expert."
    }
  ],
  "tasks": [
    {
      "title": "Write migration scripts",
      "agent": "db-specialist",
      "description": "Create SQL migrations for the new schema"
    }
  ]
}
```

When this plan executes:
1. Polpo creates `db-specialist` as a volatile agent (marked `volatile: true`, `planGroup: "data-migration"`)
2. The agent works on its tasks like any other agent
3. When the plan completes, the volatile agent is automatically removed

### Volatile Team Settings

| Setting | Type | Default | Description |
|---------|------|---------|-------------|
| `enableVolatileTeams` | `boolean` | `true` | Whether plans can define their own agents |
| `volatileCleanup` | `"on_complete" \| "manual"` | `"on_complete"` | When to remove volatile agents |

Set `volatileCleanup: "manual"` if you want volatile agents to stick around after plan completion (useful for debugging).

## Agent Activity

While an agent is working, Polpo tracks its activity in real time:

| Field | Description |
|-------|-------------|
| `lastTool` | Last tool the agent called |
| `lastFile` | Last file the agent touched |
| `filesCreated` | List of files created during this task |
| `filesEdited` | List of files edited during this task |
| `toolCalls` | Total number of tool calls so far |
| `totalTokens` | Total tokens consumed |
| `summary` | Agent's current status summary |
| `sessionId` | Session ID for adapters that support persistent sessions |

This data is visible in the TUI, the web dashboard, and the HTTP API (via `/api/state`).

## Default Config

When you initialize a new Polpo project (`polpo init`), it generates a minimal config with one default agent:

```json
{
  "project": "my-project",
  "team": {
    "name": "default",
    "description": "Default Polpo team",
    "agents": [
      { "name": "dev-1", "role": "developer" }
    ]
  },
  "settings": { }
}
```

From there, add more agents, switch adapters, or define plans with volatile teams. The octopus is flexible.
