Polpo's adapter pattern lets you integrate any agent system. This guide walks through building a custom adapter.

## The Interface

An adapter implements two things: `AgentAdapter` and `AgentHandle`.

```typescript
interface AgentAdapter {
  readonly name: string;
  spawn(agent: AgentConfig, task: Task, cwd: string): AgentHandle;
}

interface AgentHandle {
  agentName: string;
  taskId: string;
  startedAt: string;
  pid: number;                  // use 0 if not a real process
  sessionId?: string;           // optional session tracking
  activity: AgentActivity;      // live activity data
  done: Promise<TaskResult>;    // resolves when agent finishes
  isAlive(): boolean;
  kill(): void;
}
```

## Step-by-Step Example

Let's build an adapter that wraps a hypothetical `my-agent` CLI tool.

### 1. Create the Adapter File

```typescript
// src/adapters/my-agent.ts
import { spawn } from "node:child_process";
import type { AgentAdapter, AgentHandle } from "../core/adapter.js";
import type { AgentConfig, AgentActivity, Task, TaskResult } from "../core/types.js";
import { registerAdapter } from "../adapter-registry.js";

class MyAgentAdapter implements AgentAdapter {
  readonly name = "my-agent";

  spawn(agent: AgentConfig, task: Task, cwd: string): AgentHandle {
    const activity: AgentActivity = {
      filesCreated: [],
      filesEdited: [],
      toolCalls: 0,
      lastUpdate: new Date().toISOString(),
    };

    let alive = true;

    // Spawn the CLI process
    const proc = spawn("my-agent", ["--task", task.description], {
      cwd,
      stdio: ["pipe", "pipe", "pipe"],
    });

    // Track activity from stdout
    let stdout = "";
    proc.stdout.on("data", (data) => {
      stdout += data.toString();
      activity.lastUpdate = new Date().toISOString();
      activity.summary = data.toString().trim().slice(-100);
    });

    let stderr = "";
    proc.stderr.on("data", (data) => {
      stderr += data.toString();
    });

    // Create the done promise
    const done = new Promise<TaskResult>((resolve) => {
      proc.on("exit", (code) => {
        alive = false;
        resolve({
          exitCode: code ?? 1,
          stdout,
          stderr,
          duration: Date.now() - new Date(handle.startedAt).getTime(),
        });
      });
    });

    const handle: AgentHandle = {
      agentName: agent.name,
      taskId: task.id,
      startedAt: new Date().toISOString(),
      pid: proc.pid ?? 0,
      activity,
      done,
      isAlive: () => alive,
      kill: () => {
        if (alive) proc.kill("SIGTERM");
      },
    };

    return handle;
  }
}

// Self-register the adapter
registerAdapter("my-agent", new MyAgentAdapter());
```

### 2. Register via Side-Effect Import

Add the import to the CLI entry point:

```typescript
// src/cli/index.ts
import "../adapters/claude-sdk.js";
import "../adapters/generic.js";
import "../adapters/my-agent.js"; // your adapter
```

### 3. Use in Configuration

```json
{
  "agents": [
    {
      "name": "my-specialist",
      "adapter": "my-agent",
      "description": "Specialist using the my-agent framework"
    }
  ]
}
```

## Key Responsibilities

Your adapter must:

1. **Start the agent** — launch whatever process/API call is needed
2. **Pass the task** — send the task description to the agent
3. **Track activity** — update `activity` fields as the agent works
4. **Resolve `done`** — the promise must resolve with `TaskResult` when finished
5. **Implement `kill()`** — allow the orchestrator to terminate the agent
6. **Implement `isAlive()`** — return whether the agent is still running

## Activity Tracking

The `AgentActivity` object is read by the TUI and web UI to show live progress:

```typescript
interface AgentActivity {
  lastTool?: string;        // e.g. "Write", "Edit", "Bash"
  lastFile?: string;        // last file path touched
  filesCreated: string[];   // files created during this task
  filesEdited: string[];    // files edited during this task
  toolCalls: number;        // total tool calls count
  lastUpdate: string;       // ISO timestamp of last activity
  summary?: string;         // latest output/message text
  sessionId?: string;       // for transcript access
}
```

Update these fields in real-time as your agent works. The orchestrator reads them for health checks (stale detection) and the UI displays them.

## Tips

- **PID tracking**: Set `pid` to the real process ID if your agent runs as a subprocess. This enables the orchestrator to detect orphaned processes and reconnect after a crash.

- **Session ID**: If your agent supports session persistence, set `sessionId` so the `/inspect` command can read transcripts.

- **Exit codes**: Use exit code 0 for success, non-zero for failure. The orchestrator uses this to determine the initial pass/fail status before running assessment.

- **Stderr for errors**: Write error details to stderr. Assessment uses this to generate targeted fix feedback.
