Agents need tools to do their job. Polpo gives every agent a set of **coding tools** for reading, writing, and searching code — plus the ability to connect to external **MCP servers** for additional capabilities.

## Coding Tools

Polpo ships 7 built-in coding tools. They're the same primitives you'd expect from any code editor, designed for agents:

| Tool | What it does | Output limit |
|------|-------------|--------------|
| **`read`** | Read file contents with line numbers. Supports `offset` / `limit` for pagination. | 500 lines |
| **`write`** | Create or overwrite a file. Auto-creates parent directories. | — |
| **`edit`** | Search-and-replace in a file. `old_text` must match exactly once. | — |
| **`bash`** | Execute a shell command with a configurable timeout (default 120s). | 30 KB |
| **`glob`** | Find files matching a glob pattern (e.g. `**/*.ts`). | 200 results |
| **`grep`** | Regex search across files. Returns matching lines with file paths and line numbers. | 100 matches |
| **`ls`** | List a directory. Directories are suffixed with `/`. | — |

All tools except `bash` enforce the **filesystem sandbox** (see below). The `bash` tool runs commands from the project working directory.

### Restricting Tools

By default, agents get all 7 tools. You can restrict this per agent:

```json
{
  "name": "read-only-reviewer",
  "role": "code reviewer",
  "allowedTools": ["read", "glob", "grep", "ls"]
}
```

This agent can browse code but can't write, edit, or run commands — useful for review-only roles.

## Filesystem Sandbox

Agents are sandboxed by default to the **project root directory**. They can't read or write files outside it. You can tighten this further with `allowedPaths`:

```json
{
  "name": "frontend-dev",
  "allowedPaths": ["./src/ui", "./public", "./tests/ui"]
}
```

Now this agent can only touch files in those three directories (and their subdirectories). Any attempt to access `./src/api/` or `../` will be rejected with a clear error.

This is path-prefix matching with separator awareness — `./src/ui` allows `./src/ui/Button.tsx` but not `./src/ui-legacy/old.js`.

### How It Works

1. Paths are resolved to absolute paths at tool creation time
2. When no `allowedPaths` are set, the default is `[cwd]` (project root)
3. Every file operation checks `isPathAllowed()` before proceeding
4. Violations throw a descriptive error that tells the agent what it tried and where it's allowed

## Safe Environment

When agents run shell commands (`bash` tool) or spawn MCP servers, Polpo filters the environment variables to prevent leaking API keys and secrets.

The `safeEnv()` function only passes through system-essential variables:

- System: `PATH`, `HOME`, `USER`, `SHELL`, `TERM`, `LANG`, `NODE_ENV`
- Git: `GIT_AUTHOR_NAME`, `GIT_COMMITTER_NAME`, `GIT_SSH_COMMAND`, etc.
- Network: `HTTP_PROXY`, `HTTPS_PROXY`, `NO_PROXY`
- SSH: `SSH_AUTH_SOCK`

Everything else — `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, database passwords, custom secrets — is stripped. This means agent subprocesses can't exfiltrate your secrets via tool calls.

See [Security](/features/security) for more details on environment filtering.

## MCP (Model Context Protocol)

Agents can connect to external [MCP](/https://modelcontextprotocol.io/) servers for additional capabilities — databases, APIs, custom tools, anything with an MCP interface.

<Note>
Polpo is an MCP **client**, not a server. When you configure `mcpServers` on an agent, you're telling Polpo which external MCP servers to connect to. The agent discovers tools from those servers and can call them alongside the built-in coding tools.
</Note>

Polpo supports all three MCP transport types:

<Tabs>
  <Tab title="stdio">
    ```json
    {
      "name": "dev-with-db",
      "mcpServers": {
        "postgres": {
          "command": "npx",
          "args": ["-y", "@modelcontextprotocol/server-postgres", "postgresql://localhost/mydb"],
          "env": { "NODE_ENV": "development" }
        }
      }
    }
    ```
    The server is spawned as a child process using `safeEnv()` — your API keys won't leak to the MCP server process.
  </Tab>

  <Tab title="SSE">
    ```json
    {
      "name": "dev-with-remote-tools",
      "mcpServers": {
        "my-tools": {
          "type": "sse",
          "url": "https://my-mcp-server.example.com/sse",
          "headers": { "Authorization": "Bearer token123" }
        }
      }
    }
    ```
  </Tab>

  <Tab title="HTTP (Streamable)">
    ```json
    {
      "name": "dev-with-http-tools",
      "mcpServers": {
        "my-tools": {
          "type": "http",
          "url": "https://my-mcp-server.example.com/mcp",
          "headers": { "Authorization": "Bearer token123" }
        }
      }
    }
    ```
  </Tab>
</Tabs>

Tools discovered from connected MCP servers are added to the agent's tool set alongside the built-in coding tools. The agent doesn't know (or care) whether a tool is built-in or comes from an MCP server — it just uses what's available.
