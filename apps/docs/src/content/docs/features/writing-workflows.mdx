This guide walks you through creating a custom workflow from scratch. If you haven't read the [Workflows](/features/workflows/) page yet, start there for the big picture.

## Quick Start

<Steps>

1. **Create the workflow directory**

   ```bash
   mkdir -p .polpo/workflows/my-workflow
   ```

2. **Create `workflow.json`**

   ```json
   {
     "name": "my-workflow",
     "description": "What this workflow does in one sentence",
     "parameters": [
       {
         "name": "target",
         "description": "The module or file to operate on",
         "type": "string",
         "required": true
       }
     ],
     "plan": {
       "tasks": [
         {
           "title": "Do the thing to {{target}}",
           "description": "Detailed instructions for the agent about what to do with {{target}}.",
           "assignTo": "default"
         }
       ]
     }
   }
   ```

3. **Validate it**

   ```bash
   polpo workflow validate my-workflow
   ```

4. **Run it**

   ```bash
   polpo workflow run my-workflow --param target=src/auth
   ```

</Steps>

That's the whole process. The rest of this guide covers best practices and advanced patterns.

## Workflow Structure

Every workflow has three required fields and one optional:

```json
{
  "name": "kebab-case-name",
  "description": "Human-readable description",
  "parameters": [],
  "plan": {}
}
```

The `name` must match the directory name. The `plan` field is a standard [PlanDocument](/concepts/plans/) — teams, tasks, dependencies, expectations, quality gates — with `{{placeholder}}` tokens for parameterization.

## Designing Parameters

Good parameters make workflows flexible without making them complicated. Some guidelines:

**Make the minimum required.** Only mark parameters `required` if the workflow literally cannot function without them. Everything else should have a sensible `default`.

**Use enums for constrained choices.** If a parameter only makes sense with specific values, declare them:

```json
{
  "name": "severity",
  "description": "Minimum severity to flag",
  "type": "string",
  "default": "medium",
  "enum": ["low", "medium", "high", "critical"]
}
```

**Use descriptive names.** The parameter `name` appears in CLI output, TUI pickers, and error messages. `test_command` is better than `tc`.

**Document defaults.** The `description` should mention what happens when the parameter isn't provided:

```json
{
  "name": "test_command",
  "description": "Command to run tests (defaults to 'pnpm run test')",
  "type": "string",
  "default": "pnpm run test"
}
```

## Writing the Plan Template

The `plan` field is where the real work happens. It uses the same structure as a regular Plan, but with `{{placeholders}}` wherever you want parameter substitution.

### Teams

Define volatile agents specific to this workflow:

```json
{
  "plan": {
    "team": [
      {
        "name": "analyst",
        "role": "Senior code analyst — identifies issues and improvement opportunities",
        "enableGit": true
      },
      {
        "name": "fixer",
        "role": "Implementation specialist — applies targeted fixes",
        "enableGit": true
      }
    ],
    "tasks": [ ... ]
  }
}
```

<Aside type="tip">
You can omit the `team` field entirely if you want the workflow to use the project's existing agents. Just reference them by name in `assignTo`.
</Aside>

### Tasks with Placeholders

Placeholders work in any string value within the plan:

```json
{
  "title": "Analyze {{module}}",
  "description": "Review {{module}} with focus on {{focus}}.",
  "assignTo": "analyst",
  "dependsOn": ["Setup {{module}}"],
  "expectations": [
    { "type": "test", "command": "{{test_command}}" }
  ]
}
```

<Aside type="caution">
Placeholders in `dependsOn` must match the **resolved** title of the dependency. If you have a task titled `"Analyze {{module}}"` and `module=src/auth`, the dependency should be `"Analyze {{module}}"` (with the placeholder) — not `"Analyze src/auth"`. Both sides get substituted at the same time.
</Aside>

### Dependencies

Task dependencies work the same as in regular Plans — reference tasks by title:

```json
{
  "tasks": [
    {
      "title": "Diagnose {{issue}}",
      "assignTo": "debugger"
    },
    {
      "title": "Fix {{issue}}",
      "assignTo": "debugger",
      "dependsOn": ["Diagnose {{issue}}"]
    },
    {
      "title": "Verify fix",
      "assignTo": "verifier",
      "dependsOn": ["Fix {{issue}}"]
    }
  ]
}
```

### Expectations

Add assessment criteria to ensure quality:

```json
{
  "expectations": [
    {
      "type": "file_exists",
      "paths": ["analysis-report.md"]
    },
    {
      "type": "test",
      "command": "{{test_command}}"
    },
    {
      "type": "llm_review",
      "criteria": "The analysis is thorough with specific file:line references",
      "threshold": 3.5
    }
  ]
}
```

See [Assessment & Quality](/features/assessment/) for all expectation types.

### Quality Gates

Block progression until quality thresholds are met:

```json
{
  "plan": {
    "qualityGates": [
      {
        "name": "analysis-quality",
        "afterTasks": ["Analyze {{module}}"],
        "blocksTasks": ["Apply fixes"],
        "minScore": 3.5
      }
    ],
    "tasks": [ ... ]
  }
}
```

## Example: Full Workflow

Here's a complete workflow that performs a focused code review:

```json
{
  "name": "code-review",
  "description": "Deep code review of a module: analysis, fixes, and verification",
  "parameters": [
    {
      "name": "module",
      "description": "Module or directory to review",
      "type": "string",
      "required": true
    },
    {
      "name": "test_command",
      "description": "Command to run tests",
      "type": "string",
      "default": "pnpm run test"
    },
    {
      "name": "focus",
      "description": "Review focus area",
      "type": "string",
      "default": "general",
      "enum": ["general", "performance", "security", "readability", "types"]
    }
  ],
  "plan": {
    "team": [
      {
        "name": "analyst",
        "role": "Senior code analyst — identifies issues, patterns, and improvement opportunities",
        "enableGit": true
      },
      {
        "name": "reviewer",
        "role": "QA reviewer — verifies changes don't introduce regressions",
        "enableGit": true
      }
    ],
    "tasks": [
      {
        "title": "Analyze {{module}}",
        "description": "Perform a deep {{focus}} review of the {{module}} module. Identify code smells, complexity hotspots, pattern violations, type safety issues, and improvement opportunities. Write findings to analysis-report.md with specific file:line references.",
        "assignTo": "analyst",
        "expectations": [
          { "type": "file_exists", "paths": ["analysis-report.md"] },
          {
            "type": "llm_review",
            "criteria": "The analysis is thorough, covers all key aspects, and provides actionable recommendations with specific file:line references",
            "threshold": 3.5
          }
        ]
      },
      {
        "title": "Apply critical fixes",
        "description": "Based on the analysis report, apply the most critical fixes to {{module}}. Focus on: correctness bugs, type safety issues, and clear anti-patterns. Do NOT refactor for style — only fix real problems.",
        "assignTo": "analyst",
        "dependsOn": ["Analyze {{module}}"],
        "expectations": [
          { "type": "test", "command": "{{test_command}}" }
        ]
      },
      {
        "title": "Verify changes",
        "description": "Review all changes applied to {{module}}. Verify: (1) each fix addresses a real issue from the analysis, (2) no regressions introduced, (3) code quality improved. Write a brief verification summary.",
        "assignTo": "reviewer",
        "dependsOn": ["Apply critical fixes"],
        "expectations": [
          {
            "type": "llm_review",
            "criteria": "The verification confirms all changes are correct, no regressions were introduced, and code quality improved",
            "threshold": 4.0
          }
        ]
      }
    ],
    "qualityGates": [
      {
        "name": "analysis-quality",
        "afterTasks": ["Analyze {{module}}"],
        "blocksTasks": ["Apply critical fixes"],
        "minScore": 3.5
      }
    ]
  }
}
```

## Validation

Always validate before running:

```bash
polpo workflow validate code-review
```

The validator checks:

- Required fields (`name`, `description`, `plan`) are present
- The plan has at least one task
- All `{{placeholders}}` in the plan have matching parameter declarations
- All declared parameters are actually used somewhere in the plan

## Sharing Workflows

### Project-level

Put workflows in `.polpo/workflows/` — they're committed to version control and shared with your team.

### User-level

Put workflows in `~/.polpo/workflows/` — they're available across all your projects. Good for personal productivity workflows.

### Override precedence

If a project and user-level workflow share the same name, the project-level one wins. This lets teams override a personal workflow with a standardized version.

## Tips

- **Start simple.** A workflow with one task and one parameter is still useful. You can add complexity later.
- **Use quality gates for multi-step workflows.** They prevent wasted work when an early step produces poor results.
- **Write detailed task descriptions.** The agent only sees the resolved description — make it clear what "success" looks like.
- **Test with `--dry-run`.** The CLI's `--dry-run` flag shows the fully instantiated plan without executing, so you can verify placeholders resolve correctly.
- **Validate after editing.** Run `polpo workflow validate <name>` to catch placeholder/parameter mismatches early.
