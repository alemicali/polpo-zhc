The Bridge system enables Polpo to discover and monitor agent sessions that were started **outside** the orchestrator — for example, a developer running Claude Code directly from their terminal.

## Use Case

In teams where developers use AI coding agents interactively alongside orchestrated workflows, the Bridge provides visibility into those external sessions without requiring the developer to change their workflow. It answers: "What are my agents doing right now, even the ones I started manually?"

## Architecture

The Bridge is composed of three components:

```
BridgeManager (facade)
     │
     ├── SessionWatcher
     │     └── Polls filesystem for transcript files
     │         (~/.claude/projects/<encoded-cwd>/*.jsonl)
     │
     └── SessionTracker
           └── Parses transcripts, manages session lifecycle
               (active → idle → completed)
```

### BridgeManager

The top-level facade that composes SessionWatcher and SessionTracker. It can work standalone (creates its own event emitter) or be wired into an existing orchestrator's emitter for integrated mode.

### SessionWatcher

A polling-based filesystem scanner that looks for JSONL transcript files. Since JSONL files are append-only, the watcher tracks file sizes — a size increase means new content was appended (new activity).

The watcher scans:
- `~/.claude/projects/<encoded-cwd>/*.jsonl` (when `watch.claudeCode` is enabled)
- Any additional paths specified in `watch.paths`

It emits two internal events:
- `file:discovered` — a new transcript file appeared
- `file:updated` — an existing file grew (new content appended)

### SessionTracker

Maintains state for all discovered sessions. It receives file events from SessionWatcher, parses transcript summaries, and manages the session lifecycle:

| Status | Description |
|--------|-------------|
| `active` | Transcript file was recently updated |
| `idle` | No activity for half the session timeout |
| `completed` | No activity for the full session timeout |

The tracker periodically checks all sessions for inactivity. When a session exceeds the `sessionTimeout`, it transitions to `completed`.

## Configuration

```json
{
  "bridge": {
    "enabled": true,
    "mode": "read-only",
    "watch": {
      "claudeCode": true,
      "opencode": false,
      "paths": []
    },
    "pollInterval": 5000,
    "sessionTimeout": 60000
  }
}
```

### Configuration Properties

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `enabled` | `boolean` | `true` | Enable/disable the bridge |
| `mode` | `string` | `"read-only"` | Bridge operating mode |
| `watch.claudeCode` | `boolean` | `true` | Scan `~/.claude/projects/` for sessions |
| `watch.opencode` | `boolean` | `false` | Scan for OpenCode sessions |
| `watch.paths` | `string[]` | `[]` | Additional directories to scan |
| `pollInterval` | `number` | `5000` | Filesystem polling interval in ms |
| `sessionTimeout` | `number` | `60000` | Inactivity threshold before marking completed (ms) |

## Project Directory Encoding

Claude Code encodes project directories in its filesystem structure. For example, `/home/user/my-project` becomes `-home-user-my-project`. The SessionTracker decodes this back to a readable path for display.

## Session State

Each discovered session tracks:

| Field | Type | Description |
|-------|------|-------------|
| `sessionId` | `string` | Unique session identifier |
| `transcriptPath` | `string` | Full path to the JSONL transcript file |
| `projectPath` | `string` | Decoded project directory |
| `status` | `"active" \| "idle" \| "completed"` | Current session status |
| `discoveredAt` | `number` | Timestamp of discovery |
| `lastActivityAt` | `number` | Timestamp of last file change |
| `lastFileSize` | `number` | Last known transcript file size |
| `summary` | `SessionSummary?` | Parsed transcript summary |

The session summary (when available) includes message count, tool calls, files created/edited, and the last message content.

## Events

| Event | Payload | Description |
|-------|---------|-------------|
| `bridge:session:discovered` | `{ sessionId, projectPath, transcriptPath }` | A new session was found |
| `bridge:session:activity` | `{ sessionId, projectPath, messageCount, toolCalls, filesCreated, filesEdited, lastMessage }` | Session had new activity |
| `bridge:session:completed` | `{ sessionId, projectPath, summary, duration }` | Session went inactive and was marked completed |

## Starting the Bridge

The BridgeManager exposes `start()` and `stop()` methods:

```typescript
const bridge = new BridgeManager({
  pollInterval: 5000,
  sessionTimeout: 60000,
  watch: { claudeCode: true, opencode: false, paths: [] }
});

bridge.start();

// Get current sessions
const sessions = bridge.getSessions();

// Get aggregate stats
const stats = bridge.getStats();
// { total: 3, active: 1, idle: 1, completed: 1 }
```

When integrated with the orchestrator, bridge events are forwarded to the orchestrator's event emitter, making them available to the notification system, TUI, and API.

<Aside type="note">
The Bridge operates in **read-only** mode. It observes external sessions but does not interfere with them — no task creation, no agent control. It provides observability only.
</Aside>
