Polpo is configured through a JSON file at `.polpo/polpo.json` in your project directory.

## File Structure

```json
{
  "project": "my-project",
  "team": {
    "name": "default",
    "description": "Project team description",
    "agents": [
      {
        "name": "agent-name",
        "adapter": "native",
        "role": "Role description",
        "model": "opencode:big-pickle"
      }
    ]
  },
  "providers": {
    "anthropic": "${ANTHROPIC_API_KEY}",
    "openai": "${OPENAI_API_KEY}"
  },
  "settings": {
    "maxRetries": 3,
    "workDir": ".",
    "logLevel": "normal"
  },
  "approvalGates": {
    "enabled": false,
    "defaultTimeout": 3600000,
    "autoRejectOnTimeout": true
  },
  "notifications": {
    "enabled": false,
    "channels": []
  },
  "escalationPolicy": {
    "enabled": false,
    "maxEscalations": 3,
    "rules": []
  },
  "sla": {
    "enabled": false,
    "warningThreshold": 0.8,
    "violationThreshold": 1.0
  },
  "enableScheduler": false,
  "defaultQualityThreshold": 3.0
}
```

<Aside type="note">
Plans and tasks are managed dynamically via the TUI/CLI/API and stored in `.polpo/plans/*.json` and task stores — they are NOT defined in the config file.
</Aside>

## Agents

Each agent entry defines a worker that can be assigned tasks:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | yes | Unique identifier for the agent |
| `adapter` | string | no | `"native"` (default), `"claude-sdk"`, or `"generic"` |
| `role` | string | no | Agent's role and expertise description |
| `model` | string | native only | LLM model (e.g., `opencode:big-pickle`, `anthropic:claude-sonnet-4-5`) |
| `command` | string | generic only | CLI command to execute |
| `volatile` | boolean | no | Auto-cleanup when plan completes (default: `false`) |

```json
{
  "team": {
    "agents": [
      {
        "name": "developer",
        "adapter": "native",
        "model": "opencode:big-pickle",
        "role": "Full-stack developer using Polpo's native engine"
      },
      {
        "name": "architect",
        "adapter": "claude-sdk",
        "model": "claude-sonnet-4-5-20250929",
        "role": "Senior software architect who designs system components"
      },
      {
        "name": "aider-dev",
        "adapter": "generic",
        "command": "aider --no-auto-commit --model gpt-4",
        "role": "Aider-powered developer for code changes"
      }
    ]
  }
}
```

## Plans

Plans group related tasks into execution units:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `group` | string | yes | Unique plan identifier |
| `team` | Agent[] | no | Volatile agents scoped to this plan |
| `tasks` | Task[] | yes | List of tasks to execute |
| `schedule` | string | no | Cron expression for scheduled execution |
| `recurring` | boolean | no | Whether the plan re-runs on each schedule trigger (default: `false`) |
| `qualityGates` | QualityGate[] | no | Quality checkpoints that block progression |
| `qualityThreshold` | number | no | Minimum average score to pass quality gates (default: `3.0`) |
| `deadline` | string | no | ISO-8601 deadline for the plan, tracked by SLA monitoring |

### Volatile Teams

Agents defined in the `team` section are automatically created when the plan starts and cleaned up when it completes:

```json
{
  "plans": [
    {
      "group": "feature-auth",
      "team": [
        {
          "name": "auth-specialist",
          "adapter": "claude-sdk",
          "description": "Authentication expert",
          "volatile": true
        }
      ],
      "tasks": [
        {
          "title": "Implement JWT auth",
          "agent": "auth-specialist",
          "description": "Build JWT authentication middleware"
        }
      ]
    }
  ]
}
```

## Tasks

Each task is a unit of work assigned to an agent:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `title` | string | yes | Short task title |
| `agent` | string | yes | Agent name to assign to |
| `description` | string | yes | Detailed instructions for the agent |
| `dependsOn` | string[] | no | Task titles that must complete first |
| `expectations` | Expectation[] | no | Custom assessment criteria |
| `deadline` | string | no | ISO-8601 deadline for this task, tracked by SLA monitoring |
| `priority` | number | no | Task priority (lower = higher priority, default: `100`) |

## Expectations

Define how completed tasks are assessed:

```json
{
  "tasks": [
    {
      "title": "Build API endpoints",
      "agent": "backend-dev",
      "description": "Create REST API for user management",
      "expectations": [
        {
          "type": "test",
          "command": "npm test",
          "weight": 0.4
        },
        {
          "type": "file_exists",
          "path": "src/routes/users.ts",
          "weight": 0.2
        },
        {
          "type": "llm_review",
          "criteria": "Check for proper error handling and input validation",
          "weight": 0.4,
          "threshold": 3.5,
          "dimensions": [
            {
              "name": "security",
              "weight": 0.5,
              "rubric": "No SQL injection or XSS vulnerabilities"
            },
            {
              "name": "error_handling",
              "weight": 0.5,
              "rubric": "All error paths return proper HTTP status codes"
            }
          ]
        }
      ]
    }
  ]
}
```

See [Assessment](/concepts/assessment/) for details on the scoring system.

## Approval Gates

Control whether tasks require approval before proceeding:

```json
{
  "approvalGates": {
    "enabled": true,
    "defaultTimeout": 3600000,
    "autoRejectOnTimeout": true
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `enabled` | boolean | Enable the approval gate system |
| `defaultTimeout` | number | Milliseconds before an unapproved task times out (default: 1 hour) |
| `autoRejectOnTimeout` | boolean | Automatically reject tasks that exceed the timeout |

## Notifications

Configure notification channels for events:

```json
{
  "notifications": {
    "enabled": true,
    "channels": [
      {
        "type": "webhook",
        "url": "https://hooks.slack.com/services/...",
        "events": ["task:complete", "task:fail", "quality:sla"]
      }
    ]
  }
}
```

## Escalation Policy

Define escalation rules for repeated failures:

```json
{
  "escalationPolicy": {
    "enabled": true,
    "maxEscalations": 3,
    "rules": [
      {
        "afterFailures": 2,
        "action": "reassign",
        "targetAgent": "senior-dev"
      },
      {
        "afterFailures": 4,
        "action": "notify",
        "channel": "webhook"
      }
    ]
  }
}
```

## SLA Monitoring

Track deadlines on plans and tasks:

```json
{
  "sla": {
    "enabled": true,
    "warningThreshold": 0.8,
    "violationThreshold": 1.0
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `enabled` | boolean | Enable SLA deadline tracking |
| `warningThreshold` | number | Fraction of deadline elapsed before warning (default: `0.8`) |
| `violationThreshold` | number | Fraction of deadline elapsed before violation (default: `1.0`) |

## Scheduler

Enable cron-based plan scheduling:

```json
{
  "enableScheduler": true
}
```

Plans with a `schedule` field will be automatically triggered according to their cron expression.

## Default Quality Threshold

Set the global minimum score for quality gates:

```json
{
  "defaultQualityThreshold": 3.5
}
```

Individual plans can override this with their own `qualityThreshold` field.

## Environment Variables

```bash
# Required for Claude agents
ANTHROPIC_API_KEY=sk-ant-...

# Optional: customize orchestrator LLM
ORCHESTRATOR_MODEL=claude-sonnet-4-5-20250929
ORCHESTRATOR_TEMPERATURE=0.7

# Optional: server authentication
API_KEY=your-secret-key
```

## Config Path

By default, Polpo looks for `.polpo/polpo.json` in the current directory. Use `-d` or `--dir` to specify a different working directory:

```bash
polpo run -d /path/to/workdir
polpo -d /path/to/workdir
```

The config file must always be at `.polpo/polpo.json` relative to the working directory.
