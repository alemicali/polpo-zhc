This guide walks you through creating a custom template from scratch. If you haven't read the [Templates](/features/templates) page yet, start there for the big picture.

## Quick Start

<Steps>

1. **Create the template directory**

   ```bash
   mkdir -p .polpo/templates/my-template
   ```

2. **Create `template.json`**

   ```json
   {
     "name": "my-template",
     "description": "What this template does in one sentence",
     "parameters": [
       {
         "name": "target",
         "description": "The module or file to operate on",
         "type": "string",
         "required": true
       }
     ],
      "mission": {
        "tasks": [
          {
            "title": "Do the thing to {{target}}",
            "description": "Detailed instructions for the agent about what to do with {{target}}.",
            "assignTo": "default"
          }
        ]
      }
   }
   ```

3. **Validate it**

   ```bash
   polpo template validate my-template
   ```

4. **Run it**

   ```bash
   polpo template run my-template --param target=src/auth
   ```

</Steps>

That's the whole process. The rest of this guide covers best practices and advanced patterns.

## Template Structure

Every template has three required fields and one optional:

```json
{
  "name": "kebab-case-name",
  "description": "Human-readable description",
  "parameters": [],
  "mission": {}
}
```

The `name` must match the directory name. The `mission` field is a standard [MissionDocument](/concepts/missions) — teams, tasks, dependencies, expectations, quality gates — with `{{placeholder}}` tokens for parameterization.

## Designing Parameters

Good parameters make templates flexible without making them complicated. Some guidelines:

**Make the minimum required.** Only mark parameters `required` if the template literally cannot function without them. Everything else should have a sensible `default`.

**Use enums for constrained choices.** If a parameter only makes sense with specific values, declare them:

```json
{
  "name": "severity",
  "description": "Minimum severity to flag",
  "type": "string",
  "default": "medium",
  "enum": ["low", "medium", "high", "critical"]
}
```

**Use descriptive names.** The parameter `name` appears in CLI output, TUI pickers, and error messages. `test_command` is better than `tc`.

**Document defaults.** The `description` should mention what happens when the parameter isn't provided:

```json
{
  "name": "test_command",
  "description": "Command to run tests (defaults to 'pnpm run test')",
  "type": "string",
  "default": "pnpm run test"
}
```

## Writing the Mission Template

The `mission` field is where the real work happens. It uses the same structure as a regular Mission, but with `{{placeholders}}` wherever you want parameter substitution.

### Teams

Define volatile agents specific to this template:

```json
{
  "mission": {
    "team": [
      {
        "name": "analyst",
        "role": "Senior code analyst — identifies issues and improvement opportunities"
      },
      {
        "name": "fixer",
        "role": "Implementation specialist — applies targeted fixes"
      }
    ],
    "tasks": [ ... ]
  }
}
```

<Tip>
You can omit the `team` field entirely if you want the template to use the project's existing agents. Just reference them by name in `assignTo`.
</Tip>

### Tasks with Placeholders

Placeholders work in any string value within the mission:

```json
{
  "title": "Analyze {{module}}",
  "description": "Review {{module}} with focus on {{focus}}.",
  "assignTo": "analyst",
  "dependsOn": ["Setup {{module}}"],
  "expectations": [
    { "type": "test", "command": "{{test_command}}" }
  ]
}
```

<Warning>
Placeholders in `dependsOn` must match the **resolved** title of the dependency. If you have a task titled `"Analyze {{module}}"` and `module=src/auth`, the dependency should be `"Analyze {{module}}"` (with the placeholder) — not `"Analyze src/auth"`. Both sides get substituted at the same time.
</Warning>

### Dependencies

Task dependencies work the same as in regular Missions — reference tasks by title:

```json
{
  "tasks": [
    {
      "title": "Diagnose {{issue}}",
      "assignTo": "debugger"
    },
    {
      "title": "Fix {{issue}}",
      "assignTo": "debugger",
      "dependsOn": ["Diagnose {{issue}}"]
    },
    {
      "title": "Verify fix",
      "assignTo": "verifier",
      "dependsOn": ["Fix {{issue}}"]
    }
  ]
}
```

### Expectations

Add assessment criteria to ensure quality:

```json
{
  "expectations": [
    {
      "type": "file_exists",
      "paths": ["analysis-report.md"]
    },
    {
      "type": "test",
      "command": "{{test_command}}"
    },
    {
      "type": "llm_review",
      "criteria": "The analysis is thorough with specific file:line references",
      "threshold": 3.5
    }
  ]
}
```

See [Review](/features/review) for all expectation types.

### Quality Gates

Block progression until quality thresholds are met:

```json
{
  "mission": {
    "qualityGates": [
      {
        "name": "analysis-quality",
        "afterTasks": ["Analyze {{module}}"],
        "blocksTasks": ["Apply fixes"],
        "minScore": 3.5
      }
    ],
    "tasks": [ ... ]
  }
}
```

## Example: Full Template

Here's a complete template that performs a focused code review:

```json
{
  "name": "code-review",
  "description": "Deep code review of a module: analysis, fixes, and verification",
  "parameters": [
    {
      "name": "module",
      "description": "Module or directory to review",
      "type": "string",
      "required": true
    },
    {
      "name": "test_command",
      "description": "Command to run tests",
      "type": "string",
      "default": "pnpm run test"
    },
    {
      "name": "focus",
      "description": "Review focus area",
      "type": "string",
      "default": "general",
      "enum": ["general", "performance", "security", "readability", "types"]
    }
  ],
  "mission": {
    "team": [
      {
        "name": "analyst",
        "role": "Senior code analyst — identifies issues, patterns, and improvement opportunities"
      },
      {
        "name": "reviewer",
        "role": "QA reviewer — verifies changes don't introduce regressions"
      }
    ],
    "tasks": [
      {
        "title": "Analyze {{module}}",
        "description": "Perform a deep {{focus}} review of the {{module}} module. Identify code smells, complexity hotspots, pattern violations, type safety issues, and improvement opportunities. Write findings to analysis-report.md with specific file:line references.",
        "assignTo": "analyst",
        "expectations": [
          { "type": "file_exists", "paths": ["analysis-report.md"] },
          {
            "type": "llm_review",
            "criteria": "The analysis is thorough, covers all key aspects, and provides actionable recommendations with specific file:line references",
            "threshold": 3.5
          }
        ]
      },
      {
        "title": "Apply critical fixes",
        "description": "Based on the analysis report, apply the most critical fixes to {{module}}. Focus on: correctness bugs, type safety issues, and clear anti-patterns. Do NOT refactor for style — only fix real problems.",
        "assignTo": "analyst",
        "dependsOn": ["Analyze {{module}}"],
        "expectations": [
          { "type": "test", "command": "{{test_command}}" }
        ]
      },
      {
        "title": "Verify changes",
        "description": "Review all changes applied to {{module}}. Verify: (1) each fix addresses a real issue from the analysis, (2) no regressions introduced, (3) code quality improved. Write a brief verification summary.",
        "assignTo": "reviewer",
        "dependsOn": ["Apply critical fixes"],
        "expectations": [
          {
            "type": "llm_review",
            "criteria": "The verification confirms all changes are correct, no regressions were introduced, and code quality improved",
            "threshold": 4.0
          }
        ]
      }
    ],
    "qualityGates": [
      {
        "name": "analysis-quality",
        "afterTasks": ["Analyze {{module}}"],
        "blocksTasks": ["Apply critical fixes"],
        "minScore": 3.5
      }
    ]
  }
}
```

## Validation

Always validate before running:

```bash
polpo template validate code-review
```

The validator checks:

- Required fields (`name`, `description`, `mission`) are present and valid
- Name is kebab-case (e.g. `bug-fix`, `code-review`)
- All `{{placeholders}}` in the mission have matching parameter declarations
- Optional parameters without a `default` value are not used as `{{placeholders}}` in the mission (would fail at runtime if omitted)
- Parameter definitions have valid `name`, `description`, and `type` fields

## Sharing Templates

### Project-level

Put templates in `.polpo/templates/` — they're committed to version control and shared with your team.

### User-level

Put templates in `~/.polpo/templates/` — they're available across all your projects. Good for personal productivity templates.

### Override precedence

If a project and user-level template share the same name, the project-level one wins. This lets teams override a personal template with a standardized version.

## Tips

- **Start simple.** A template with one task and one parameter is still useful. You can add complexity later.
- **Use quality gates for multi-step templates.** They prevent wasted work when an early step produces poor results.
- **Write detailed task descriptions.** The agent only sees the resolved description — make it clear what "success" looks like.
- **Test with `--dry-run`.** The CLI's `--dry-run` flag shows the fully instantiated mission without executing, so you can verify placeholders resolve correctly.
- **Validate after editing.** Run `polpo template validate <name>` to catch placeholder/parameter mismatches early.
