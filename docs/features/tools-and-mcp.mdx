Agents need tools to do their job. Polpo gives every agent a set of **core tools** for reading, writing, searching files, and making HTTP requests — **5 extended tool categories** (browser, email, vault, image, and video) — plus the ability to connect to external **MCP servers** for additional capabilities.

## Core Tools

Polpo ships 9 built-in core tools. They're the same primitives you'd expect from any code editor, designed for agents — plus HTTP tools for making requests and downloading files:

| Tool | What it does | Output limit |
|------|-------------|--------------|
| **`read`** | Read file contents with line numbers. Supports `offset` / `limit` for pagination. | 500 lines |
| **`write`** | Create or overwrite a file. Auto-creates parent directories. | — |
| **`edit`** | Search-and-replace in a file. `old_text` must match exactly once. | — |
| **`bash`** | Execute a shell command with a configurable timeout (default 120s). | 30 KB |
| **`glob`** | Find files matching a glob pattern (e.g. `**/*.ts`). | 200 results |
| **`grep`** | Regex search across files. Returns matching lines with file paths and line numbers. | 100 matches |
| **`ls`** | List a directory. Directories are suffixed with `/`. | — |
| **`http_fetch`** | Send an HTTP request and return the response. | 100 KB |
| **`http_download`** | Download a file from a URL and save it to disk. | — |

All tools except `bash` enforce the **filesystem sandbox** (see below). The `bash` tool runs commands from the project working directory.

<Info>
For full parameter schemas, return types, error codes, and usage examples for each core tool, see the [Core Tools reference](/reference/coding-tools). For HTTP tool details, see the [HTTP tools reference](/reference/tools/http).
</Info>

### Restricting Tools

By default, agents get all 9 core tools. You can restrict this per agent:

```json
{
  "name": "read-only-reviewer",
  "role": "code reviewer",
  "allowedTools": ["read", "glob", "grep", "ls"]
}
```

This agent can browse code but can't write, edit, or run commands — useful for review-only roles.

## Extended Tools

Beyond the 9 core tools, Polpo offers **5 optional tool categories** that can be enabled per-agent via the `allowedTools` array.

<Note>
**Simplified toolset**: Git, multi-file operations, dependency management, Excel/CSV, PDF, Word/DOCX, and audio tools have been removed as built-in tool categories. Agents should use the `bash` tool with appropriate CLI commands or install [skills](/features/skills) for these operations instead. HTTP tools (`http_fetch`, `http_download`) are now always-on core tools — no flag needed.
</Note>

### Browser (18 tools)

Automate browser interactions — navigate, click, fill forms, take screenshots, extract data. Powered by the `agent-browser` CLI.

```json
{ "name": "qa-agent", "allowedTools": ["read", "write", "edit", "bash", "glob", "grep", "ls", "browser_*"] }
```

Key tools: `browser_navigate`, `browser_click`, `browser_type`, `browser_screenshot`, `browser_evaluate`, `browser_wait`, `browser_select`, `browser_scroll`, `browser_drag_drop`, and more.

For authenticated browsing, set up a persistent profile with `polpo browser login <agent> <url>` — the agent reuses the login session across tasks. See [Browser tools reference](/reference/tools/browser#persistent-browser-profiles) for details.

### Email (5 tools)

Send, read, and search emails via SMTP and IMAP — useful for communication agents, notification bots, or inbox monitoring.

```json
{ "name": "alice", "allowedTools": ["read", "write", "edit", "bash", "glob", "grep", "ls", "email_*"] }
```

Tools: `email_send`, `email_verify`, `email_list`, `email_read`, `email_search`.

Email tools require SMTP/IMAP credentials. The easiest way to set them up is through the [onboarding wizard](/guides/build-your-team#onboard-your-agents) (`polpo agent onboard alice`), which creates vault entries with `${ENV_VAR}` references. Credentials can also be set via environment variables (`SMTP_HOST`, `IMAP_HOST`, etc.) or tool parameter overrides.

### Vault (2 tools)

Read-only access to the agent's own vault credentials at runtime — essential for skills or tasks that need API keys, tokens, or custom credentials beyond the built-in SMTP/IMAP flow (which is automatic).

```json
{ "name": "integrations-agent", "allowedTools": ["read", "write", "edit", "bash", "glob", "grep", "ls", "vault_*"] }
```

Tools: `vault_list` (list available services with masked values), `vault_get` (retrieve actual credential values for a service).

Credentials are stored in the encrypted vault (`.polpo/vault.enc`) and resolved at spawn time. The agent can only see its own entries — never other agents' credentials.

### Image (3 tools)

Generate images from text prompts and analyze existing images with vision models. Image generation is powered by **fal.ai** (FLUX models), while vision/analysis uses OpenAI or Anthropic.

```json
{ "name": "designer", "allowedTools": ["read", "write", "edit", "bash", "glob", "grep", "ls", "image_*"] }
```

Tools: `image_generate` (text-to-image via fal.ai FLUX), `image_analyze` (vision/OCR via OpenAI GPT-4.1-mini or Anthropic Claude).

Image generation requires the `FAL_KEY` environment variable. Image analysis requires `OPENAI_API_KEY` or `ANTHROPIC_API_KEY`.

### Video (1 tool)

Generate videos from text prompts using **fal.ai** (Wan 2.2 models).

```json
{ "name": "video-creator", "allowedTools": ["read", "write", "edit", "bash", "glob", "grep", "ls", "video_*"] }
```

Tools: `video_generate` (text-to-video via fal.ai Wan 2.2).

Video generation requires the `FAL_KEY` environment variable. Generation takes 1-5 minutes depending on model and resolution.

### Enabling Multiple Categories

You can enable multiple extended categories on a single agent:

```json
{
  "name": "outreach-agent",
  "allowedTools": ["read", "write", "edit", "bash", "glob", "grep", "ls", "browser_*", "email_*", "vault_*", "image_*", "video_*"]
}
```

## Filesystem Sandbox

Agents are sandboxed by default to the **project root directory**. They can't read or write files outside it. You can tighten this further with `allowedPaths`:

```json
{
  "name": "frontend-dev",
  "allowedPaths": ["./src/ui", "./public", "./tests/ui"]
}
```

Now this agent can only touch files in those three directories (and their subdirectories). Any attempt to access `./src/api/` or `../` will be rejected with a clear error.

This is path-prefix matching with separator awareness — `./src/ui` allows `./src/ui/Button.tsx` but not `./src/ui-legacy/old.js`.

### How It Works

1. Paths are resolved to absolute paths at tool creation time
2. When no `allowedPaths` are set, the default is `[cwd]` (project root)
3. Every file operation checks `isPathAllowed()` before proceeding
4. Violations throw a descriptive error that tells the agent what it tried and where it's allowed

## Safe Environment

When agents run shell commands (`bash` tool) or spawn MCP servers, Polpo filters the environment variables to prevent leaking API keys and secrets.

The `safeEnv()` function only passes through system-essential variables:

- System: `PATH`, `HOME`, `USER`, `SHELL`, `TERM`, `LANG`, `NODE_ENV`
- Git: `GIT_AUTHOR_NAME`, `GIT_COMMITTER_NAME`, `GIT_SSH_COMMAND`, etc.
- Network: `HTTP_PROXY`, `HTTPS_PROXY`, `NO_PROXY`
- SSH: `SSH_AUTH_SOCK`

Everything else — `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, database passwords, custom secrets — is stripped. This means agent subprocesses can't exfiltrate your secrets via tool calls.

See [Security](/features/security) for more details on environment filtering.

## MCP (Model Context Protocol)

Agents can connect to external [MCP](https://modelcontextprotocol.io/) servers for additional capabilities — databases, APIs, custom tools, anything with an MCP interface.

<Note>
Polpo is an MCP **client**, not a server. When you configure `mcpServers` on an agent, you're telling Polpo which external MCP servers to connect to. The agent discovers tools from those servers and can call them alongside the built-in core tools.
</Note>

Polpo supports all three MCP transport types:

<Tabs>
  <Tab title="stdio">
    ```json
    {
      "name": "dev-with-db",
      "mcpServers": {
        "postgres": {
          "command": "npx",
          "args": ["-y", "@modelcontextprotocol/server-postgres", "postgresql://localhost/mydb"],
          "env": { "NODE_ENV": "development" }
        }
      }
    }
    ```
    The server is spawned as a child process using `safeEnv()` — your API keys won't leak to the MCP server process.
  </Tab>

  <Tab title="SSE">
    ```json
    {
      "name": "dev-with-remote-tools",
      "mcpServers": {
        "my-tools": {
          "type": "sse",
          "url": "https://my-mcp-server.example.com/sse",
          "headers": { "Authorization": "Bearer token123" }
        }
      }
    }
    ```
  </Tab>

  <Tab title="HTTP (Streamable)">
    ```json
    {
      "name": "dev-with-http-tools",
      "mcpServers": {
        "my-tools": {
          "type": "http",
          "url": "https://my-mcp-server.example.com/mcp",
          "headers": { "Authorization": "Bearer token123" }
        }
      }
    }
    ```
  </Tab>
</Tabs>

Tools discovered from connected MCP servers are added to the agent's tool set alongside the built-in core tools. The agent doesn't know (or care) whether a tool is built-in or comes from an MCP server — it just uses what's available.
