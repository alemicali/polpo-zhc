title: "Memory"

Polpo has three persistent knowledge stores: **shared memory** for all agents, **agent memory** for individual agents, and **system context** for Polpo itself.

## Shared Memory

Shared memory is a markdown file (`.polpo/memory.md`) that Polpo injects into every agent's context. It's persistent across sessions and visible to all agents working on your project.

Use it to capture:
- Architecture decisions and patterns
- Build commands and common operations
- Known issues and workarounds
- Coding style and conventions
- Project-specific knowledge

## Agent Memory

Agent memory gives each agent its own private memory space. Stored in `.polpo/memory/<agent-name>.md` (file backend) or as `agent:<name>` keys in the database.

Use it to capture:
- Agent-specific patterns and preferences
- Lessons learned from past tasks
- Tool-specific knowledge
- Coding style particular to that agent's domain

When an agent is spawned, it receives **both** the shared memory and its own private memory.

## Creating Memory

### Via CLI

```bash
# Shared memory
polpo memory show
polpo memory set "# Architecture\nTypeScript + React"
polpo memory append "Use vitest for all tests"
polpo memory edit

# Agent memory
polpo memory show --agent frontend-dev
polpo memory set --agent frontend-dev "Prefers functional components"
polpo memory append --agent frontend-dev "Learned to use React Server Components"
polpo memory edit --agent frontend-dev
```

### Via TUI

```
/memory                         # View shared memory
/memory edit                    # Edit shared memory
/memory agent:frontend-dev      # View agent memory
/memory agent:frontend-dev edit # Edit agent memory
```

### Via Web UI

Navigate to **Memory** in the sidebar. Use the **Scope** selector to switch between shared memory and any agent's private memory.

### Via API

```bash
# Shared memory
curl http://localhost:3000/api/memory
curl -X PUT http://localhost:3000/api/memory \
  -H "Content-Type: application/json" \
  -d '{"content": "# Shared Memory\n\n## Build\n./node_modules/.bin/tsc"}'

# Agent memory
curl http://localhost:3000/api/memory/agent/frontend-dev
curl -X PUT http://localhost:3000/api/memory/agent/frontend-dev \
  -H "Content-Type: application/json" \
  -d '{"content": "Prefers tailwind for styling"}'
```

### Via Orchestrator Tools

The orchestrator can read and write both shared and agent memory:

```
get_memory                    # Read shared memory
get_memory(agent: "alice")    # Read agent memory
save_memory(content: "...")   # Write shared memory
save_memory(content: "...", agent: "alice")  # Write agent memory
append_memory(content: "...") # Append to shared memory
append_memory(content: "...", agent: "alice") # Append to agent memory
```

### Manually

Create `.polpo/memory.md` for shared memory:

```markdown
# Shared Memory

## Build
- Run `./node_modules/.bin/tsc` to compile (not `npx tsc`)
- Run `npm test` to verify

## Architecture
- Express + SQLite backend
- React 18 frontend with shadcn/ui
```

Create `.polpo/memory/<agent-name>.md` for agent memory:

```markdown
# Frontend Dev Notes

## Patterns
- Always use functional components with hooks
- Prefer server components where possible
- Use Zod for form validation
```

## How Agents Use Memory

When an agent is spawned for a task, Polpo:

1. Reads `.polpo/memory.md` (shared memory)
2. Reads `.polpo/memory/<agent-name>.md` (agent-specific memory, if it exists)
3. Injects both into the task description using XML tags:
   - `<shared-memory>` — visible to all agents
   - `<agent-memory>` — private to the assigned agent

This means every agent automatically knows about shared project context, plus has access to its own accumulated knowledge.

## Best Practices

1. **Keep shared memory concise** — focus on project-wide facts (build commands, architecture, conventions)
2. **Use agent memory for specialization** — tool preferences, domain-specific patterns, lessons learned
3. **Update frequently** — add new learnings as agents discover them
4. **Be specific** — "Use `./node_modules/.bin/tsc`" is better than "use the TypeScript compiler"
5. **Include commands** — exact build, test, and lint commands save agent time
6. **Note known issues** — prevents agents from rediscovering the same problems

<Note>
Memory is available via the React SDK's `useMemory()` hook for shared memory and `useAgentMemory(agentName)` for agent-specific memory.
</Note>

## System Context

System context (`.polpo/system-context.md`) is a separate file that stores **standing instructions for Polpo itself** — not for the agents it spawns. Every time you start a conversation with Polpo (via TUI or the `/v1/chat/completions` endpoint), this file is injected into Polpo's system prompt.

Use it for:
- Standing instructions ("always use TypeScript strict mode")
- Project conventions Polpo should follow when generating missions
- Preferences for agent selection or model choice
- Anything you'd tell a project manager on day one

### How It Works

1. Polpo reads `.polpo/system-context.md` at the start of every conversation
2. The content is injected as "Standing instructions" in Polpo's system prompt
3. Polpo follows these instructions when planning, delegating, and reviewing work

### Creating System Context

Tell Polpo to remember something during a conversation:

```
> Remember: always run npm test after any code change
```

Polpo uses its `append_system_context` tool to write this to `.polpo/system-context.md`. You can also create or edit the file manually.

### Memory vs System Context

| | Shared Memory | Agent Memory | System Context |
|---|---|---|---|
| **File** | `.polpo/memory.md` | `.polpo/memory/<agent>.md` | `.polpo/system-context.md` |
| **Injected into** | All agent prompts | Assigned agent's prompt | Polpo's own prompt |
| **Purpose** | Project knowledge for all agents | Private knowledge per agent | Standing instructions for Polpo |
| **Who uses it** | All worker agents | Individual agent | Polpo (the manager) |
| **Managed via** | `/memory`, API, CLI, or manual edit | `/memory --agent`, API, CLI, or manual edit | `append_system_context` tool, or manual edit |
