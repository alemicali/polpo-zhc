title: "Templates"
description: "Parameterized, reusable plan templates — define a process once, run it with different parameters."
---

Templates are **parameterized plan templates** — JSON files that define a complete [Plan](/concepts/plans) with `{{placeholder}}` parameters. When you execute a template, Polpo substitutes your parameters, saves the result as a standard Plan, and runs it.

```bash
# Review the auth module, focusing on security
polpo template run code-review --param module=src/auth --param focus=security

# Fix a specific bug
polpo template run bug-fix --param issue="Login fails when email has + character"
```

A template is data, not code — a JSON file on disk. When executed, it becomes a regular Plan with no new runtime concepts or special execution path.

## Quick Start

<Steps>

1. **Create the template directory**

   ```bash
   mkdir -p .polpo/templates/my-template
   ```

2. **Create `template.json`**

   ```json
   {
     "name": "my-template",
     "description": "What this template does in one sentence",
     "parameters": [
       {
         "name": "target",
         "description": "The module or file to operate on",
         "type": "string",
         "required": true
       }
     ],
     "plan": {
       "tasks": [
         {
           "title": "Do the thing to {{target}}",
           "description": "Detailed instructions for the agent about what to do with {{target}}.",
           "assignTo": "default"
         }
       ]
     }
   }
   ```

3. **Validate it**

   ```bash
   polpo template validate my-template
   ```

4. **Run it**

   ```bash
   polpo template run my-template --param target=src/auth
   ```

</Steps>

## Anatomy of a Template

A template lives in a directory with a `template.json` file:

```
.polpo/templates/
  code-review/
    template.json
  bug-fix/
    template.json
```

The `template.json` has three required fields and one optional:

```json
{
  "name": "code-review",
  "description": "Deep code review of a module: analysis, fixes, and verification",
  "parameters": [ ... ],
  "plan": { ... }
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | Yes | Template identifier (kebab-case, must match directory name) |
| `description` | string | Yes | Human-readable summary |
| `parameters` | array | No | Declared parameters (see below) |
| `plan` | object | Yes | Plan body — same shape as a PlanDocument, with `{{placeholders}}` |

## Parameters

Each parameter declares a name, type, and constraints:

```json
{
  "name": "module",
  "description": "Module or directory to review",
  "type": "string",
  "required": true
}
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `name` | string | — | Parameter name, used as `{{name}}` in the plan |
| `description` | string | — | Human-readable explanation |
| `type` | `"string"` \| `"number"` \| `"boolean"` | `"string"` | Value type — booleans accept `true/false/yes/no/1/0` |
| `required` | boolean | `false` | Whether the parameter must be provided |
| `default` | string \| number \| boolean | — | Default value when not provided |
| `enum` | array | — | Allowed values (enum constraint) |

<Tip>
Parameters with a `default` value don't need to be marked `required` — they'll always resolve to something.
</Tip>

### Enum Parameters

Constrain a parameter to specific values:

```json
{
  "name": "focus",
  "description": "Review focus area",
  "type": "string",
  "default": "general",
  "enum": ["general", "performance", "security", "readability", "types"]
}
```

### Parameter Design Tips

- **Make the minimum required.** Only mark parameters `required` if the template literally cannot function without them. Everything else should have a sensible `default`.
- **Use enums for constrained choices.** If a parameter only makes sense with specific values, declare them.
- **Use descriptive names.** The parameter `name` appears in CLI output, TUI pickers, and error messages. `test_command` is better than `tc`.
- **Document defaults in `description`.** Mention what happens when the parameter isn't provided:

```json
{
  "name": "test_command",
  "description": "Command to run tests (defaults to 'pnpm run test')",
  "type": "string",
  "default": "pnpm run test"
}
```

## Writing the Plan Template

The `plan` field is a standard Plan structure — teams, tasks, dependencies, expectations, quality gates — with `{{placeholder}}` tokens wherever you want parameter substitution.

### Teams

Define volatile agents specific to this template:

```json
{
  "plan": {
    "team": [
      {
        "name": "analyst",
        "role": "Senior code analyst — identifies issues and improvement opportunities",
        "enableGit": true
      },
      {
        "name": "fixer",
        "role": "Implementation specialist — applies targeted fixes",
        "enableGit": true
      }
    ],
    "tasks": [ ... ]
  }
}
```

<Tip>
You can omit the `team` field entirely if you want the template to use the project's existing agents. Just reference them by name in `assignTo`.
</Tip>

### Tasks with Placeholders

Placeholders work in any string value within the plan:

```json
{
  "title": "Analyze {{module}}",
  "description": "Review {{module}} with focus on {{focus}}.",
  "assignTo": "analyst",
  "dependsOn": ["Setup {{module}}"],
  "expectations": [
    { "type": "test", "command": "{{test_command}}" }
  ]
}
```

Placeholders work anywhere in the plan JSON — titles, descriptions, commands, file paths, even `dependsOn` references. During instantiation, Polpo serializes the plan to a string, replaces all `{{param}}` tokens, and parses it back.

<Note>
Every placeholder in the plan **must** correspond to a declared parameter (or have a default). Polpo will throw an error if any `{{placeholder}}` remains unreplaced after substitution.
</Note>

### Dependencies

Task dependencies work the same as in regular Plans — reference tasks by title:

```json
{
  "tasks": [
    {
      "title": "Diagnose {{issue}}",
      "assignTo": "debugger"
    },
    {
      "title": "Fix {{issue}}",
      "assignTo": "debugger",
      "dependsOn": ["Diagnose {{issue}}"]
    },
    {
      "title": "Verify fix",
      "assignTo": "verifier",
      "dependsOn": ["Fix {{issue}}"]
    }
  ]
}
```

<Warning>
Placeholders in `dependsOn` must match the **template** title of the dependency — use `"Diagnose {{issue}}"`, not the resolved `"Diagnose Login bug"`. Both sides get substituted at the same time.
</Warning>

### Expectations

Add assessment criteria to ensure quality:

```json
{
  "expectations": [
    {
      "type": "file_exists",
      "paths": ["analysis-report.md"]
    },
    {
      "type": "test",
      "command": "{{test_command}}"
    },
    {
      "type": "llm_review",
      "criteria": "The analysis is thorough with specific file:line references",
      "threshold": 3.5
    }
  ]
}
```

See [Review](/features/review) for all expectation types.

### Quality Gates

Block progression until quality thresholds are met:

```json
{
  "plan": {
    "qualityGates": [
      {
        "name": "analysis-quality",
        "afterTasks": ["Analyze {{module}}"],
        "blocksTasks": ["Apply fixes"],
        "minScore": 3.5
      }
    ],
    "tasks": [ ... ]
  }
}
```

## Discovery

Polpo discovers templates from three locations, in priority order (first match wins by name):

| Priority | Location | Scope |
|----------|----------|-------|
| 1 | `<polpoDir>/templates/` | Project-level |
| 2 | `<cwd>/.polpo/templates/` | Fallback (if polpoDir differs) |
| 3 | `~/.polpo/templates/` | User-level (shared across projects) |

Each template is a **directory** containing a `template.json` file. Directories without a valid `template.json` are silently skipped.

If two templates share the same `name`, the higher-priority location wins. This lets you override a user-level template with a project-specific version.

## Execution Flow

When you run a template, four things happen:

1. **Load** — the template definition is loaded from disk
2. **Validate** — parameters are checked against declared types, enums, and required flags. Defaults are applied.
3. **Instantiate** — all `{{placeholder}}` tokens are replaced with resolved values. The result is validated as legal JSON.
4. **Execute** — the instantiated plan is saved via `savePlan()` and executed via `executePlan()` — from here on it's a standard Plan

```
template.json + params → validation → substitution → PlanDocument → executePlan()
```

The template system adds zero runtime overhead — once instantiated, the plan executes through the exact same path as any manually created plan. Same task lifecycle, same assessment, same quality gates, same hooks.

## Validation

Always validate before running:

```bash
polpo template validate code-review
```

The validator checks:

- Required fields (`name`, `description`, `plan`) are present
- The plan has at least one task
- All `{{placeholders}}` in the plan have matching parameter declarations
- All declared parameters are actually used somewhere in the plan

## Example: Full Template

Here's a complete template that performs a focused code review:

```json
{
  "name": "code-review",
  "description": "Deep code review of a module: analysis, fixes, and verification",
  "parameters": [
    {
      "name": "module",
      "description": "Module or directory to review",
      "type": "string",
      "required": true
    },
    {
      "name": "test_command",
      "description": "Command to run tests",
      "type": "string",
      "default": "pnpm run test"
    },
    {
      "name": "focus",
      "description": "Review focus area",
      "type": "string",
      "default": "general",
      "enum": ["general", "performance", "security", "readability", "types"]
    }
  ],
  "plan": {
    "team": [
      {
        "name": "analyst",
        "role": "Senior code analyst — identifies issues, patterns, and improvement opportunities",
        "enableGit": true
      },
      {
        "name": "reviewer",
        "role": "QA reviewer — verifies changes don't introduce regressions",
        "enableGit": true
      }
    ],
    "tasks": [
      {
        "title": "Analyze {{module}}",
        "description": "Perform a deep {{focus}} review of the {{module}} module. Identify code smells, complexity hotspots, pattern violations, type safety issues, and improvement opportunities. Write findings to analysis-report.md with specific file:line references.",
        "assignTo": "analyst",
        "expectations": [
          { "type": "file_exists", "paths": ["analysis-report.md"] },
          {
            "type": "llm_review",
            "criteria": "The analysis is thorough, covers all key aspects, and provides actionable recommendations with specific file:line references",
            "threshold": 3.5
          }
        ]
      },
      {
        "title": "Apply critical fixes",
        "description": "Based on the analysis report, apply the most critical fixes to {{module}}. Focus on: correctness bugs, type safety issues, and clear anti-patterns. Do NOT refactor for style — only fix real problems.",
        "assignTo": "analyst",
        "dependsOn": ["Analyze {{module}}"],
        "expectations": [
          { "type": "test", "command": "{{test_command}}" }
        ]
      },
      {
        "title": "Verify changes",
        "description": "Review all changes applied to {{module}}. Verify: (1) each fix addresses a real issue from the analysis, (2) no regressions introduced, (3) code quality improved. Write a brief verification summary.",
        "assignTo": "reviewer",
        "dependsOn": ["Apply critical fixes"],
        "expectations": [
          {
            "type": "llm_review",
            "criteria": "The verification confirms all changes are correct, no regressions were introduced, and code quality improved",
            "threshold": 4.0
          }
        ]
      }
    ],
    "qualityGates": [
      {
        "name": "analysis-quality",
        "afterTasks": ["Analyze {{module}}"],
        "blocksTasks": ["Apply critical fixes"],
        "minScore": 3.5
      }
    ]
  }
}
```

## Built-in Templates

Polpo ships with 8 ready-to-use templates:

| Template | Description | Key Parameters |
|----------|-------------|----------------|
| `code-review` | Deep code review with analysis, fixes, and verification | `module` (required), `focus`, `test_command` |
| `bug-fix` | Structured bug fix: reproduce, fix, test, verify | `issue` (required), `scope`, `test_command` |
| `refactoring` | Safe refactoring with analysis and test validation | `module` (required), `goal` (required), `test_command` |
| `test-coverage` | Improve test coverage for a module | `module` (required), `target_coverage`, `test_command` |
| `documentation` | Generate or improve documentation | `module` (required), `style`, `scope` |
| `security-audit` | Security audit with vulnerability assessment | `module` (required), `severity`, `test_command` |
| `migration` | Database or API migration template | `from` (required), `to` (required), `test_command` |
| `onboarding` | Generate onboarding docs for a codebase | `module` (required), `audience` |

## Sharing Templates

### Project-level

Put templates in `.polpo/templates/` — they're committed to version control and shared with your team.

### User-level

Put templates in `~/.polpo/templates/` — they're available across all your projects. Good for personal productivity templates.

### Override precedence

If a project and user-level template share the same name, the project-level one wins. This lets teams override a personal template with a standardized version.

## Templates vs Skills vs Plans

| Dimension | Skills | Templates | Plans |
|-----------|--------|-----------|-------|
| **Layer** | Prompt | Orchestration | Execution |
| **Format** | Markdown (`SKILL.md`) | JSON (`template.json`) | JSON (`.polpo/plans/`) |
| **What it does** | Injects instructions into agent's system prompt | Generates a Plan from a parameterized template | Defines tasks, agents, and dependencies to execute |
| **Reusability** | Assigned to agents across tasks | Instantiated with different parameters | Single-use (one execution) |
| **Runtime impact** | Affects how the agent thinks | None — becomes a standard Plan | Drives task execution |

Skills make agents smarter. Templates make processes repeatable. Plans are the execution unit.

## Tips

- **Start simple.** A template with one task and one parameter is still useful. You can add complexity later.
- **Use quality gates for multi-step templates.** They prevent wasted work when an early step produces poor results.
- **Write detailed task descriptions.** The agent only sees the resolved description — make it clear what "success" looks like.
- **Test with `--dry-run`.** The CLI's `--dry-run` flag shows the fully instantiated plan without executing, so you can verify placeholders resolve correctly.
- **Validate after editing.** Run `polpo template validate <name>` to catch placeholder/parameter mismatches early.
